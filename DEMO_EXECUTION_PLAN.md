# SoothPal Demo 冲刺执行方案（Web + 小程序复用）

> 目标：做出“评委一眼觉得厉害”的高保真 Demo，先上线 GitHub Pages，再平滑复用到小程序 Demo。

## 1. 交付目标与成功标准

### 1.1 核心目标
- 产出一个可公网访问的 Web Demo（GitHub Pages）。
- 完成 3 分钟“震慑型演示”闭环：
  - 10 秒疼痛打卡（2D 身体点选+强度）
  - 语音一句话 -> AI 结构化抽取卡
  - 即时预警卡（rule_id + 原因解释）
  - 一键周报（趋势+建议+就医提示）
- 前端观感达到“产品级”：视觉统一、动效流畅、交互反馈细腻。

### 1.2 评委向验收标准
- 第一屏 3 秒内建立专业感：品牌、医疗可信、AI 深度融合。
- 任意关键动作（点击、滑动、切页）有明确反馈，不卡顿。
- 所有关键结论可解释：预警来自规则，不是“AI 乱说”。
- 演示台词与页面状态一一对应，支持现场快速切换场景。

## 2. 技术路线总览

## 2.1 Web Demo 技术栈（主战场）
- 框架：Vite + React + TypeScript
- 动效：Framer Motion + GSAP（页面过渡、卡片入场、关键反馈）
- 状态管理：Zustand（轻量、适合 Demo 场景编排）
- 图表：ECharts（趋势图、热度图）
- 样式：SCSS Modules + Design Tokens（非模板化视觉）
- 图标/插画：自定义 SVG 组件（保证风格统一）
- 构建部署：GitHub Actions -> GitHub Pages

为什么选这套：
- 对 GitHub Pages 最友好（纯静态产物）。
- 视觉与动画控制力强，适合“高冲击演示界面”。
- 与后续小程序复用更容易（业务逻辑与设计 token 可迁移）。

## 2.2 小程序 Demo 技术栈（第二阶段）
- 优先：Taro + React + TypeScript（最大化复用 Web 业务层）
- 备选：原生小程序 + 独立逻辑包（若 Taro 受限）
- 云侧：CloudBase（云函数 + 数据库 + 存储）

复用原则：
- 复用“业务模型与规则引擎配置”，不强行复用全部 UI。
- Web 做“视觉天花板”，小程序做“体验可落地”。

## 3. 官方信息核验后的保守结论（用于答辩口径）

## 3.1 微信版本与发布
- 官方文档明确开发版、体验版、审核版、线上版四类版本。
- 体验版用于内测，适合比赛现场扫码体验。
- 正式发布需走上传代码 -> 提交审核 -> 发布流程。

## 3.2 小程序备案
- 官方备案资料与流程文档存在完整操作指引。
- 备案审核常见口径为 3-20 个工作日区间。
- 方案中应使用“受类目与材料完整性影响”的表述，不给死天数承诺。

## 3.3 CloudBase 成本口径
- 官方计费页给出“基础套餐 + 按量付费”机制。
- 存在开发阶段免费体验活动（有前提和时效限制）。
- 对外统一口径：开发期优先免费体验，试点期按量测算成本。

## 3.4 IMA 口径
- 可确认 IMA 是知识库工作台产品。
- 公开页面未提供稳定对外 API 作为后端检索标准依赖。
- Demo 架构不将 IMA 作为硬依赖后端；RAG 用“可替换检索层”设计。

## 4. 信息架构与页面规划（Web Demo）

## 4.1 页面流（单页分段 + 场景切换）
1. Hero：品牌定位 + 核心价值（低负担采集、可审计预警、AI 深度融合）
2. 采集场景：2D 身体图点选 + 强度滑条 + 即时保存
3. AI 抽取场景：语音输入动画 -> 结构化 JSON 卡片
4. 风险预警场景：规则命中 -> 红色预警卡 + 原因解释
5. 周报场景：趋势图 + AI 周报摘要 + 引用标识
6. 亲情号场景：家属视角摘要通知
7. 技术底座场景：架构图 + 安全与合规说明

## 4.2 演示模式设计
- 演示控制台（仅开发模式可见）：
  - 一键切换低/中/高风险数据
  - 一键触发语音抽取成功/失败降级
  - 一键播放完整 3 分钟演示路径
- 保底策略：支持“无网络 mock 模式”。

## 5. 视觉与交互标准（前端下狠功夫）

## 5.1 视觉方向
- 风格关键词：医疗可信 + 科技质感 + 情感陪伴
- 背景：多层渐变 + 细颗粒纹理 + 微弱动态光斑
- 字体建议：
  - 标题：Source Han Sans SC / HarmonyOS Sans SC
  - 数据数字：JetBrains Mono / DIN 系
- 颜色系统（首版）：
  - Primary: #1F5EFF
  - Accent: #00B894
  - Alert: #F04438
  - Surface: #F7FAFF / #ECF3FF

## 5.2 动效策略
- 页面进入：分层 stagger reveal（不是简单淡入）
- 关键交互：
  - 点击热区有“波纹 + 吸附”反馈
  - 滑条变化实时映射颜色与表情
  - 预警弹层使用“先震动后聚焦”动画
- 性能目标：中端设备保持 50-60fps 观感

## 5.3 适配策略
- 桌面优先（评委演示主场）+ 移动端兼容
- 断点：>=1280、>=1024、>=768、<768
- 演示分辨率适配：1920x1080 与 1366x768

## 8. GitHub Pages 上线策略

- 采用 GitHub Actions 自定义工作流部署。
- `main` 分支提交后自动构建并发布到 Pages。
- 保留 `demo-stable` 标签版本作为比赛应急回滚点。

部署注意：
- 项目型 Pages 需处理 `base` 路径。
- 所有资源走相对路径，避免 404。
- 输出目录使用 `dist`，由 workflow 上传为 pages artifact。
