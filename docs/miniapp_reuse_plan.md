# Web Demo -> 小程序 Demo 复用迁移方案

## 1. 迁移目标

- 保留 Web Demo 的核心业务闭环：采集、抽取、预警、周报、亲情号。
- 最大化复用模型与规则，减少重复开发。
- 小程序端以“稳定体验”为先，不强求 1:1 视觉还原。

## 2. 推荐路线

- 框架优先：Taro + React + TypeScript
- 云侧优先：CloudBase（云函数 + 数据库 + 存储）
- 发布形态：体验版二维码（比赛阶段）

## 3. 可复用资产

## 3.1 直接复用
- 数据模型：`PainEntry`, `RiskEvent`, `WeeklyReport`
- 风险规则配置：阈值、突增规则、状态机映射
- AI 抽取 schema：字段定义与校验逻辑
- 演示脚本：场景切换与话术

## 3.2 部分复用
- 页面文案与结构：可复用信息架构，但 UI 组件需重写
- 图表配置：ECharts 配置逻辑可迁移，小程序端改为兼容库

## 3.3 不建议复用
- Web 大量 GSAP 动效
- 复杂背景层与高成本滤镜

## 4. 建议的共享包拆分

```text
shared-contracts/
  types/
    pain.ts
    risk.ts
    report.ts
  rules/
    risk-rules.json
  ai/
    extract-schema.json
    guardrail-prompts.ts
```

双端依赖策略：
- Web 与小程序都依赖 `shared-contracts`
- 各端只实现自己的 UI 与平台适配层

## 5. 小程序适配重点

- 输入：语音权限、录音时长、失败重试机制
- 图表：替换为小程序图表组件
- 推送：订阅消息模板与触达频控
- 隐私：最小化采集 + 明示同意 + 服务类目对齐

## 6. 开发排期（7 天）

1. Day 1：初始化 Taro 工程，接入共享模型。
2. Day 2：完成 Scene A（打卡）页面与本地状态。
3. Day 3：完成 Scene B（抽取）与云函数 mock。
4. Day 4：完成 Scene C（预警）规则落地。
5. Day 5：完成 Scene D（周报）与亲情号摘要页。
6. Day 6：联调 + 体验版发布 + 演示脚本适配。
7. Day 7：性能优化 + 现场预案演练。

## 7. 风险与规避

- 风险：小程序渲染性能不如 Web
  - 对策：减少高开销动画，优先交互反馈
- 风险：审核与备案节奏不可控
  - 对策：比赛阶段仅使用体验版，不依赖公开上架
- 风险：外部 AI 能力波动
  - 对策：抽取与周报保留 mock 回退策略

## 8. 验收标准

- 评委扫码后 10 秒内完成首次操作
- 可稳定演示至少 5 分钟无崩溃
- 所有高危预警都能展示触发依据
